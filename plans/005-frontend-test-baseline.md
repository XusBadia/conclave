## Why this matters

The desktop frontend is 12.3K lines of TypeScript/React with **zero** tests
and **no test runner installed** — `apps/desktop/package.json` has no `test`
script at all. The only gate is `tsc -b && vite build`. Meanwhile the app's
most complex frontend logic (the 5,186-line `routes/Cases.tsx`: batch event
state machine, drag-drop, PDF export flow) is also its highest-churn file.
Nobody can safely refactor it — and a refactor is wanted (see
plans/README.md, deferred findings) — until tests exist. This plan creates
the infrastructure and seeds it with characterization tests for the pure
modules, so that (a) every future commit runs frontend tests via the
pre-commit hook, and (b) the follow-up "extract and test the batch state
machine" work has rails to land on.

Deliberately NOT in this plan: testing `Cases.tsx` itself. Its logic must be
extracted into hooks before it is testable; that extraction is a separate,
larger effort that should not be bundled with infrastructure setup.

## Steps

### Step 1: Install and configure Vitest

1. `pnpm --dir apps/desktop add -D vitest jsdom @testing-library/react @testing-library/jest-dom`
   (let pnpm resolve versions; vitest must be compatible with vite ^5 — if
   peer-dependency resolution fails, pin `vitest@^2` which targets Vite 5).
2. Create `apps/desktop/vitest.config.ts`:

   ```ts
   import { defineConfig } from "vitest/config";
   import react from "@vitejs/plugin-react";

   export default defineConfig({
     plugins: [react()],
     test: {
       environment: "jsdom",
       include: ["src/**/*.test.{ts,tsx}"],
       setupFiles: ["src/test-setup.ts"],
     },
   });
   ```

3. Create `apps/desktop/src/test-setup.ts` containing exactly:
   `import "@testing-library/jest-dom/vitest";`
4. Add to package.json scripts: `"test": "vitest run"`.
5. Smoke check: create `apps/desktop/src/lib/cn.test.ts` with one trivial
   test of the real `cn` export (read `src/lib/cn.ts` first; assert observed
   behavior, e.g. merging two classes).

**Verify**: `pnpm --dir apps/desktop test` → 1 file, 1 test, passing.
**Verify**: `pnpm --dir apps/desktop build` → still exit 0 (if `tsc -b`
errors on test files, add the minimal fix — e.g. `"types": ["vitest"]` or
excluding `vitest.config.ts` from the build tsconfig — and re-run until BOTH
commands pass).

### Step 2: Characterization tests for the PDF filename + options modules

Read `src/pdf/filename.ts` and `src/pdf/exportOptions.ts` fully, then write:

1. `src/pdf/filename.test.ts` — cover at minimum: a representative
   happy-path filename; behavior with empty/missing label inputs; behavior
   with characters that are illegal in filenames (`/`, `:`) if the
   implementation handles them (assert what it DOES, and if it does NOT
   sanitize path separators, mark the test
   `// characterization: documents current behavior — see plans/README.md`
   and note it in your report rather than fixing it here).
2. `src/pdf/exportOptions.test.ts` — `DEFAULT_PDF_EXPORT_OPTIONS` shape
   stays stable (snapshot-free: assert the keys and defaults explicitly);
   `loadPdfExportOptions` returns defaults when `localStorage` is empty,
   honors a stored value, and survives corrupted JSON in storage (wrap:
   set `localStorage.setItem(<the key the module uses>, "{nope")` and assert
   it falls back to defaults rather than throwing — if it throws, that's a
   real finding: mark the test `.skip` with a comment and report it).

**Verify**: `pnpm --dir apps/desktop test` → all pass (≥ 6 tests total).

### Step 3: Characterization tests for provider metadata helpers

Read `src/lib/providers.ts`; pick its exported pure functions (model lists,
label/formatting helpers, status mapping — whatever is exported and free of
Tauri imports) and write `src/lib/providers.test.ts` with 3–6 assertions that
pin current behavior, prioritizing anything the Settings flow branches on.
If every export depends on `ipc.ts`/Tauri, write tests only for what is pure
and say so in your report.

**Verify**: `pnpm --dir apps/desktop test` → all pass.

### Step 4: Wire into the verification gate

In `scripts/verify.sh`, after the `pnpm --dir apps/desktop build` step, add:

```sh
step "pnpm --dir apps/desktop test"
pnpm --dir apps/desktop test
```

(Use the existing `step` helper; keep ordering — build first catches type
errors with better messages, tests run on the already-built types.)

**Verify**: `./scripts/verify.sh` → all five steps run; ends "✓ All local checks passed".

## STOP conditions

Stop and report back (do not improvise) if:

- Vitest cannot be installed compatibly with Vite 5 even pinned to `^2`
  (dependency resolution error twice in a row).
- `tsc -b` cannot be made to accept the test files with a one-line tsconfig
  change — do not restructure tsconfigs.
- A module you're characterizing imports `@tauri-apps/*` at top level (would
  need mocking infrastructure this plan deliberately excludes) — drop that
  module from scope and note it.
- The verify.sh test step pushes the pre-commit wall time beyond ~6 minutes
  warm — report; the operator may prefer tests before the build step or a
  reduced scope.
